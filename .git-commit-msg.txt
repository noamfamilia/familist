Keep optimistic auth offline; persist queue history across modal hide/show

Two fixes that share a session-scoped store as the source of truth.

Drop-to-guest on offline (regression from 1187793). The optimistic local-account
boot was treating any failed Supabase verification as an authoritative server
rejection, including the offline gate inside `runSessionPipeline`, the
`INITIAL_SESSION` timeout catch, and `handleBootVerifyFailureRef`. Going offline
or refreshing the app while offline silently signed the user out of their cached
account and kicked them to guest. Now:

* `runSessionPipeline` no longer drops to guest when `!isBrowserOnline()` — it
  just exits and the existing `online` listener re-runs the pipeline on
  reconnect.
* `INITIAL_SESSION` timeout no longer drops the optimistic account boot to
  guest. The toast still surfaces when we were online so a real Supabase
  outage is still visible.
* `handleBootVerifyFailureRef` only drops when the failure is authoritative:
  `code === '450'` (refresh-token revoked) always counts, and other server
  codes only count when the browser was online and the error is not classified
  as connectivity-shaped by `isLikelyConnectivityError`. Network-shaped or
  null-session-while-offline failures keep the optimistic boot in place.

Queue history surviving modal hide/show. The Queue history list used to be
populated only by `PendingQueueStatusSection` observing the live Dexie queue,
so anything that completed while the About modal was closed was never recorded
and the list looked "cleared" when the modal reopened. Move authorship into
the sync engine via `recordQueueRowHistory`, which pre-computes the row
description before deletion and writes a display-ready entry into the shared
module-level store. The section now just subscribes and renders. The
live-query diff path is retained only as a fallback for rows that leave the
queue via coalesce/scrub paths that don't go through the drain. History
clears only on actual page reload (module state lives in JS memory).
